# v0.3 First Slice Task Decision

## 1. 候选项判断

本轮从 P2 候选项中选择一个小切片：

- 群聊主题引导
  - 能改善讨论聚焦度，但会影响提示词、群聊数据结构和 AI 回复行为。
  - 适合在界面主体验更稳定后实现。

- 思考强度配置
  - 有价值，但依赖模型供应商能力差异和 API 参数映射。
  - 容易和多 API 管理、模型调用策略耦合，不适合作为 v0.3 首个切片。

- Token 花费统计面板
  - 价值明确，但需要先记录模型调用 token、供应商返回字段和计费规则。
  - 当前没有完整调用计量基础，不适合作为首个 P2 切片。

- 白天和黑夜模式
  - 实现成本低，但对 AI 群聊核心体验提升不如消息阅读体验直接。

- 更好的桌面端与移动端适配
  - 范围较宽，容易变成无边界前端 polish。
  - 应拆到具体页面和具体问题后再做。

- 消息显示队列优化为真实聊天页面效果
  - 用户已明确提出过。
  - 直接改善当前高频主路径：阅读消息、区分用户和 AI 角色、观察 AI 自动追加回复。
  - 主要是前端模板、CSS 和少量 JS 结构调整，不需要改 AI 调用、账号隔离或 API 管理主链路。
  - 适合作为 v0.3 首个增强体验切片。

- 群聊历史检索、筛选和管理
  - 需要新增查询入口和筛选规则，价值较高但范围大于首个体验切片。

## 2. 本切片选择

本切片实现消息显示队列的聊天化呈现。

## 3. 当前目标

让群聊详情页的消息流更接近真实聊天场景：用户消息、AI 角色消息和系统消息有清晰视觉区分，新增消息自动追加后仍保持可读、稳定、可扫描。

## 4. UI 行为

- 群聊详情页消息区改为聊天流布局。
- 用户消息靠右展示。
- AI 角色消息靠左展示，并清楚显示 AI 角色名称。
- 系统消息居中或弱化展示，用于错误和状态说明。
- 每条消息以气泡或等价结构呈现。
- 消息内容不能和发送者、按钮、输入框重叠。
- 发送中、等待 AI 回复、错误状态保持可见。
- 异步追加的新消息沿用同一聊天化样式。
- 移动端和桌面端都必须能完成阅读和发送主路径。

## 5. 后端行为

- 不新增后端业务能力。
- 不改变消息保存结构。
- 不改变异步发送、轮询、AI 互评、角色权限和账号隔离规则。
- 如前端渲染需要，可只调整现有消息 JSON 到 DOM 的 class 或结构映射。

## 6. 验收检查

- 用户消息和 AI 消息能通过布局和样式明显区分。
- 不同 AI 角色的发送者名称清晰可见。
- 系统消息不被误认为用户或 AI 回复。
- 发送消息后，用户消息立即追加为正确样式。
- AI 普通回复自动追加为正确样式。
- AI 互评回复自动追加为正确样式。
- 历史消息重新进入群聊后仍以同样样式显示。
- 消息内容较长时不会溢出容器或遮挡输入区。
- 桌面端和移动端宽度下主路径可用。
- v0.2 主路径不回退：登录、账号隔离、API 管理、角色创建/编辑/删除、发言权限、异步回复、AI 互评仍可用。

## 7. 明确不做

- 不做主题引导。
- 不做思考强度配置。
- 不做 Token 花费统计。
- 不做白天/黑夜模式。
- 不做群聊历史搜索。
- 不做文件上传。
- 不做工具调用。
- 不做 WebSocket 重构。
- 不做公开多人聊天室。
- 不做完整设计系统重构。

## 8. 停止条件

- 如果聊天化展示需要改变消息存储结构，停止说明。
- 如果实现需要重写异步消息追加链路，停止说明。
- 如果实现扩大到主题、统计、文件、工具或 P3 范围，停止收窄。
- 如果样式改动导致发送、追加、角色管理或 AI 互评主路径不可用，停止修复主路径。

1. 调整群聊详情页历史消息 HTML 结构。

3. 更新 CSS，实现聊天化消息流。
4. 保持现有异步发送和轮询行为不变。
5. 增加或更新 HTTP/前端相关测试覆盖关键 class 或页面输出。
6. 执行自动测试和构建。

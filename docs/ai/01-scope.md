# Product Scope

## 1. P0：MVP 必须有，没有它就不是这个产品

- AI 群聊房间
  - 原因：产品的核心不是单人单 AI 对话，而是用户进入一个群聊空间，让多个 AI 角色围绕同一上下文参与讨论。没有群聊房间，就无法验证产品主价值。

- 群聊名称
  - 原因：用户需要能区分不同讨论场景。MVP 中只需要最基础的命名能力，不需要复杂分类、标签或搜索。

- AI 角色添加与基础配置
  - 原因：多角色是产品成立的前提。P0 只要求角色名称、人设、回复风格、模型选择这类最小配置，不加入复杂权限、头像、模板市场等扩展能力。

- 至少两个 AI 角色参与同一轮讨论
  - 原因：产品要验证“多个 AI 从不同视角讨论”的价值。P0 不要求 AI 之间复杂辩论策略，但必须能看到多个角色基于用户消息分别回复。

- 用户发送消息，AI 角色生成回复
  - 原因：这是群聊体验的最小闭环。没有用户输入和 AI 回复，就无法验证核心使用场景。

- 清晰显示发送者身份
  - 原因：用户必须区分消息来自自己还是哪个 AI 角色。否则多 AI 群聊会变成不可读的普通文本流。

- 基础模型 API 配置
  - 原因：AI 角色必须能调用模型。P0 只需要支持最小可用配置，让角色能使用一个已配置模型，不做完整 API 管理后台。

- 群聊消息保存与再次查看
  - 原因：群聊讨论天然依赖上下文和回看。P0 只要求基础持久化，不要求复杂检索、归档或长期上下文管理。

- 网页端可用界面
  - 原因：产品定位是网页端 AI 聊天群。P0 需要一个可完成核心流程的网页界面，但不追求完整视觉系统或移动端精细适配。

## 2. P1：MVP 后尽快做

- 用户登录与账号数据隔离
  - 原因：愿景中包含用户登录，但首版是否单用户还是多用户仍待确认。它重要，但不应阻塞 P0 对 AI 群聊核心价值的验证。

- 完整 API 管理
  - 原因：长期需要在总设置中管理多个 API、模型和账号绑定关系。P0 只需要最小可用配置，完整管理能力适合在核心闭环验证后补齐。

- AI 角色删除、编辑和发言权限开关
  - 原因：角色生命周期管理会很快影响可用性，但不是验证“多个 AI 群聊是否成立”的最低条件。

- 实时消息推送
  - 原因：实时体验是产品的重要方向，但具体是否必须使用 WebSocket 属于技术实现选择。P1 应补齐稳定的实时推送体验。

- AI 角色之间的补充或反驳
  - 原因：这是产品差异化能力，但规则容易复杂化。MVP 先验证多角色分别回复，再扩展角色间互动。

- MVP 级验收与基础测试
  - 原因：核心闭环跑通后，需要尽快沉淀验收标准和基础测试，避免后续功能叠加时破坏主流程。

## 3. P2：增强体验

- 群聊主题引导
  - 原因：主题能改善讨论聚焦度，但用户也可以先通过普通消息表达主题，因此不属于 P0。

- 思考强度配置
  - 原因：它能增强模型使用控制，但会增加配置复杂度。应在基础模型选择稳定后再加入。

- Token 花费统计面板
  - 原因：成本可见性很重要，但它依赖模型调用记录和计费数据。核心聊天闭环完成后再做更合理。

- 白天和黑夜模式
  - 原因：属于使用体验增强，不影响产品核心价值验证。

- 更好的桌面端与移动端适配
  - 原因：P0 只要求网页端可用。精细响应式体验适合在核心流程稳定后优化。

- 群聊历史检索、筛选和管理
  - 原因：历史保存属于 P0，但复杂管理能力不是首轮验证必需。

## 4. P3：远期功能

- 文件上传与 AI 文件分析
  - 原因：文件能力会引入存储、解析、安全和模型上下文处理问题，明显超出首轮 AI 群聊验证范围。

- 工具调用
  - 原因：工具调用涉及权限、安全边界、执行环境和结果校验。它是长期能力，不应在核心群聊尚未稳定前引入。

- 多供应商、多模型高级路由
  - 原因：多模型管理是方向，但高级路由会增加成本、故障处理和用户理解负担，适合作为远期能力。

- 公开社区或多人实时协作聊天室
  - 原因：这会把产品推向社交和协作平台，复杂度远高于个人 AI 群聊工具。

- 插件或角色市场
  - 原因：市场化能力依赖稳定的角色模型、审核机制和分享体系，应在产品核心体验成熟后考虑。

## 5. Non-goals：明确不做

- 不做通用社交 IM
  - 原因：产品目标是 AI 多角色讨论，不是替代微信、Discord 或 Slack。

- 不做公开大型聊天室
  - 原因：公开多人聊天会引入审核、反滥用、运营和社区治理问题，偏离当前产品边界。

- 不做模型训练、微调或模型托管
  - 原因：本项目使用外部或已配置模型能力，不建设底层模型平台。

- 不在 MVP 中做文件分析、工具调用和 Token 统计
  - 原因：这些能力有价值，但都会拉大范围，削弱对 AI 群聊核心闭环的验证。

- 不在 MVP 中做复杂权限体系
  - 原因：P0 只验证单个用户管理自己的群聊和 AI 角色，不处理团队、组织、共享和细粒度权限。

- 不在产品范围未确认前写业务代码
  - 原因：当前阶段目标是收敛产品边界，避免技术实现先行导致返工。

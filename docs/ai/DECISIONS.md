- 原因：用户目标是完成 roadmap 中 v1.0 稳定版，涉及多个 P3 能力、数据结构、UI、验证和文档收口。
- 影响范围：后续执行按 S0 -> S2 -> S3 -> S4 -> S5 推进，并持续维护 `STATUS.md`、`NEXT.md`、`BACKLOG.md`、`DECISIONS.md`。

## 2026-05-02

- 决策或假设：v1.0 第一个切片选择“文件上传与 AI 文件分析”的最小可运行闭环。
- 原因：它是 v1.0 roadmap 中最直接的用户可验证能力，可以复用现有群聊、消息、角色和 OpenAI-compatible 调用路径；相比工具调用、多供应商路由和 CI，能更快形成真实主路径闭环。
- 影响范围：第一切片只支持受控文本类文件上传、保存和注入 AI 上下文，不做复杂文件解析、向量检索、OCR、PDF 专项解析或工具调用。

## 2026-05-02

- 决策或假设：聊天分析文件默认保存到 `data/chat-files`，不复用公开静态 `/uploads` 目录。
- 原因：头像图片可以公开访问，但聊天分析文件可能包含用户私有内容；默认非公开目录更符合账号隔离和稳定版安全边界。
- 影响范围：新增 `CHAT_FILE_DIR` 配置；`.gitignore` 忽略 `data/`；聊天页只展示文件元数据，不提供公开下载链接。

## 2026-05-02

- 决策或假设：文件上传切片扩展支持 `.docx` 和文本型 `.pdf`，并把单文件上限从 1MB 调整到 10MB。
- 原因：用户验收反馈指出需要直接本地上传常见 Word/PDF 文档；1MB 对真实文档也过窄。
- 影响范围：上传 input 的 `accept` 范围扩大；后端新增 `.docx` 和基础 PDF 文本提取；UI 明确为从本机选择或拖拽上传；扫描版 PDF、OCR 和复杂 PDF 高保真解析仍不进入本轮。

## 2026-05-02

- 决策或假设：本地文件选择后立即自动提交上传，同时保留“上传文件”按钮作为明确状态和无脚本兜底。
- 原因：用户反馈选择文件后没有反应；对当前 UI 来说二次点击上传按钮不够明显。
- 影响范围：`chat.js` 监听文件选择和拖拽 drop 后自动提交；上传区域在黑夜模式下增加显式文字颜色。

## 2026-05-02

- 决策或假设：v1.0 工具调用第一切片只实现用户手动触发的内置白名单工具，不实现模型自动调用和任意命令执行。
- 原因：roadmap 要求受控范围、权限和失败状态清晰；直接开放 shell、外网或插件会带来高风险并扩大范围。
- 影响范围：当前只提供 `current_time`、`text_stats`、`calculator`；工具结果保存为系统消息和 `tool_executions` 记录；后续模型自动 tool calling 另行切片。

## 2026-05-03

- 决策或假设：模型 API 的 TLS/network timeout 按瞬时外部依赖故障处理，先在 OpenAI-compatible 客户端层增加有限重试和可配置超时，不在业务层吞掉失败。
- 原因：用户报告多个 AI 角色同时出现 `TLS handshake timeout`，说明问题发生在模型供应商连接阶段；客户端层统一处理可以覆盖 `/models` 和 `/chat/completions`。
- 影响范围：新增 `MODEL_API_TIMEOUT_SECONDS`、`MODEL_API_TLS_HANDSHAKE_TIMEOUT_SECONDS`、`MODEL_API_RETRY_ATTEMPTS`、`MODEL_API_RETRY_BACKOFF_MS`；最终超时错误改为中文可操作提示；异步回复背景上下文延长到 5 分钟。

- 影响范围：当前轮只验证自动测试、构建、本地服务启动、登录和群聊创建主路径；真实模型回复仍交由用户在本机浏览器检查。

## 2026-05-03

- 原因：用户当前关注 AI 回答质感，尤其互评像真人；最小风险路径是先把互评从“补充/反驳报告”改成“自然群聊接话”，避免扩大到复杂对话策略。

## 2026-05-03

- 决策或假设：用户反馈 prompt-only 优化后仍然“很人机”，根因之一是机制层固定全员回答和固定互评；本轮改为确定性的选择性参与，而不是继续只堆 prompt。
- 原因：真实群聊通常不是每个人每轮都说话，也不会每轮固定互评；减少固定模式比继续强化文案更能降低机械感。
- 影响范围：三个以上可发言角色时首轮只选两个角色回复；短消息不触发互评；互评最多追加一个跟进回复；前端轮询改为等待最小核心回复和短暂静默，不再按全员加固定互评数量等待。数据库、UI 配置和模型供应商接口不变。

## 2026-05-03

- 决策或假设：加快 AI 回答优先并发化首轮已选角色的模型调用，不并发保存消息，也不把互评提前到首轮之前。
- 原因：模型供应商调用是主要等待时间；并发请求两个首轮角色能把等待时间从两个请求相加降到接近最慢一个请求，同时按顺序保存消息可以降低 store 并发风险并保持页面展示稳定。
- 影响范围：`generateAIReplies` 并发调用 `GenerateReply`，完成后按选择顺序保存消息和记录 token；底部状态文案统一为“AI 正在回复...”，不再提前提示“随后进行互评”。真实速度仍取决于模型供应商延迟和慢的那一路响应。

## 2026-05-03

- 决策或假设：回滚“首轮只选两个 AI 角色发言”的策略，保留并发加速和最多一条互评。
- 原因：用户反馈该策略导致每次只有两个 AI 回答并卡住流程；用户当前明确期望所有允许发言角色都完成首轮回复。
- 影响范围：`selectFirstRoundRoles` 恢复返回所有可发言角色；这些角色仍并发请求模型；前端轮询恢复等待当前允许发言角色数；互评仍最多一条，短消息仍不触发互评。

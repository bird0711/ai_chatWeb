# v0.2 Third Slice Task Decision

## 1. 候选项判断

本轮从剩余 P1 候选项中选择一个小切片：

- 用户登录与账号数据隔离
  - 价值高，但会牵动用户表、会话、数据归属、现有单用户数据迁移和所有查询边界。
  - 不适合作为当前第三切片直接实现。

- 完整 API 管理
  - 价值高，但会牵动模型配置数据结构、角色模型选择、多个配置的选择规则和密钥管理。
  - 不适合作为当前第三切片直接实现。

- AI-to-AI 补充或反驳
  - 仍属于 v0.2 P1。
  - 可以复用现有群聊、角色、消息、AI 调用和异步追加链路。
  - 可以做成可开关的小切片，不重构账号体系和 API 管理。

## 2. 本切片选择

本切片实现 AI-to-AI 补充/反驳的最小版本。

## 3. 当前目标

让用户在群聊中开启“AI 互评”后，AI 角色在第一轮分别回复用户消息之后，系统再触发少量 AI 角色基于已有 AI 回复做补充或反驳，从而验证多 AI 角色之间互动的产品价值。

## 4. UI 行为

- 群聊详情页提供“AI 互评”开关。
- 默认关闭。
- 开启后，用户发送消息时：
  - 第一轮仍由允许发言的 AI 角色分别回复用户。
  - 第一轮回复完成后，再产生最多 2 条互评回复。
- 互评回复仍作为普通 AI 消息追加到当前消息流。
- 互评回复的发送者仍显示为具体 AI 角色名称。
- 页面不新增复杂策略配置。

## 5. 后端行为

- 群聊数据增加 `ai_review_enabled` 字段，旧群聊默认关闭。
- 发送消息时仍要求至少两个允许发言 AI 角色。
- 第一轮回复规则不变。
- 当 `ai_review_enabled` 开启时：
  - 至少第一轮有两个 AI 回复成功后，才触发互评。
  - 最多选择前两个允许发言角色产生互评回复。
  - 互评提示词应要求角色基于其他 AI 的回答进行补充或反驳，避免重复第一轮回答。
  - 互评消息保存为普通 AI 消息。
- 互评失败不应删除用户消息和第一轮 AI 回复。
- 如果互评失败，可写入系统消息说明失败原因。

## 6. 验收检查

- 用户可以在群聊详情页开启 AI 互评。
- 关闭 AI 互评时，发送消息只产生第一轮 AI 回复。
- 开启 AI 互评时，发送消息在第一轮 AI 回复后追加互评回复。
- 互评回复能看到具体 AI 角色名称。
- 互评回复基于已有 AI 回复进行补充或反驳，而不是只重复回答用户问题。
- 少于两个允许发言 AI 角色时仍阻止发送。
- v0.2 第一、第二切片不回退：异步发送、自动追加、群聊删除、角色删除、角色编辑和发言权限开关仍可用。
- 自动测试覆盖服务层和 HTTP 路由行为。

## 7. 明确不做

- 不做用户登录与账号隔离。
- 不做完整多 API 管理后台。
- 不做复杂 AI 互评策略配置。
- 不做无限轮 AI 互相对话。
- 不做用户点名某个 AI 回复。
- 不做文件上传、工具调用、Token 统计。
- 不做真实聊天气泡视觉优化；该项已进入 P2/v0.3。

## 8. 停止条件

- 如果实现需要引入账号体系，停止说明。
- 如果实现需要重构模型 API 配置，停止说明。
- 如果实现会破坏异步消息追加链路，停止说明。
- 如果互评逻辑扩展为多轮复杂辩论，停止并收窄为本切片范围。

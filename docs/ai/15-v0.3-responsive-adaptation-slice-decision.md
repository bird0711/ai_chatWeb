# v0.3 Responsive Adaptation Slice Decision

## 1. 背景

v0.3/P2 包含更好的桌面端与移动端适配。当前聊天详情页已经有三栏布局、侧栏菜单、消息搜索、主题、角色配置和发送区，但窄屏下仍需要更明确的阅读和发送优先级。

用户已确认思考强度配置正常，因此可以推进下一个低耦合体验切片。

## 2. 本切片选择

本切片选择“聊天详情页响应式适配增强”。

原因：

- 属于 v0.3/P2 明确范围。
- 主要改 CSS，不触碰消息存储、AI 调用或后端路由。
- 直接服务 roadmap 中“移动端至少能完成主要阅读和发送流程”的完成标准。
- 当前核心使用路径集中在聊天详情页，先优化该页面收益最高。

## 3. 当前目标

让桌面端和移动端在聊天详情页都能稳定阅读、搜索、配置角色和发送消息。窄屏下优先显示聊天主区，设置区保持在下方折叠菜单中。

## 4. 产出

- 放宽聊天主列的最小宽度，避免中间消息区撑破布局。
- 在平板宽度下将右侧设置区移动到下方网格。
- 在窄屏下让聊天主区优先展示，设置和角色区后置。
- 优化顶部导航在窄屏下的换行和账号区域显示。
- 优化消息区高度、内边距、气泡宽度和头像尺寸。
- 优化发送表单在手机宽度下的触控尺寸，并保持发送区易访问。

## 5. 内容边界

- 只做 CSS 层响应式增强。
- 不改 HTML 结构。
- 不改后端路由。
- 不改数据库。
- 不改消息发送、轮询、AI 回复、AI 互评、主题引导或模型参数。

## 6. 验收检查

- 桌面端聊天三栏布局仍可用。
- 平板宽度下消息主区不会横向溢出。
- 手机宽度下主聊天区优先显示。
- 手机宽度下发送表单按钮有足够触控高度。
- 长消息内容不撑破页面。
- 自动测试和构建通过。

## 7. 明确不做

- 不做全站视觉重设计。
- 不做原生 App 适配。
- 不做复杂可访问性审计。
- 不做性能专项。
- 不做 Token 统计。

## 8. 下一轮验证切片

由用户在本机浏览器验证桌面端和移动端或窄屏模拟下的聊天详情页阅读、搜索和发送流程是否正常。

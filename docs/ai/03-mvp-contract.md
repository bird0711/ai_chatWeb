# v0.1 MVP Implementation Contract

## 1. 本轮目标

实现 v0.1 MVP 的最小可验证产品闭环：用户在网页端创建一个 AI 群聊，配置至少两个 AI 角色，发送一条消息，看到多个 AI 角色以各自身份回复，并能在再次进入群聊时看到历史消息。

本轮目标不是后端 demo，也不是接口样例，而是一个用户可以通过网页界面完成的端到端 MVP。

## 2. 本轮必须实现的用户界面

- 群聊列表或入口页
  - 用户可以看到已创建的群聊。
  - 用户可以创建新的群聊并填写群聊名称。

- 群聊详情页
  - 用户可以查看当前群聊名称。
  - 用户可以查看消息列表。
  - 用户可以看到每条消息的发送者身份。
  - 用户可以输入并发送消息。

- AI 角色配置界面
  - 用户可以为当前群聊添加至少两个 AI 角色。
  - 用户可以为每个 AI 角色填写角色名称、人设、回复风格和模型选择。

- 基础模型 API 配置界面
  - 用户可以填写 v0.1 所需的最小模型调用配置。
  - 用户可以让 AI 角色使用该配置完成回复。

## 3. 本轮必须实现的用户操作路径

- 用户打开网页端应用。
- 用户创建一个带名称的 AI 群聊。
- 用户进入该群聊。
- 用户添加至少两个 AI 角色。
- 用户为每个 AI 角色设置名称、人设、回复风格和模型。
- 用户完成基础模型 API 配置。
- 用户在群聊中发送一条消息。
- 系统展示用户消息。
- 至少两个 AI 角色分别基于用户消息生成回复。
- 系统展示每个 AI 角色的名称和回复内容。
- 用户离开后再次进入该群聊，仍能看到历史消息。

## 4. 本轮必须实现的后端能力

- 群聊数据管理
  - 创建群聊。
  - 读取群聊列表。
  - 读取单个群聊详情。
  - 保存群聊名称。

- AI 角色数据管理
  - 为群聊添加 AI 角色。
  - 保存角色名称、人设、回复风格和模型选择。
  - 读取群聊内的 AI 角色列表。

- 基础模型 API 配置
  - 保存 v0.1 所需的最小模型调用配置。
  - 将角色配置中的模型选择用于 AI 回复。

- 消息数据管理
  - 保存用户消息。
  - 保存 AI 角色回复。
  - 按群聊读取历史消息。
  - 保留发送者身份，包括用户和 AI 角色名称。

- AI 回复编排
  - 用户发送消息后，至少触发两个 AI 角色分别生成回复。
  - 每个 AI 角色回复时必须带入该角色的人设、回复风格和当前群聊上下文。
  - AI 回复必须接入真实模型调用路径；不允许只用固定 mock 文本伪造完整闭环。

## 5. 本轮必须验证的行为

- 用户能通过网页界面创建群聊，而不是只通过后端接口创建。
- 用户能通过网页界面添加并配置至少两个 AI 角色。
- 用户能通过网页界面配置基础模型 API。
- 用户发送消息后，页面能展示用户消息和至少两个 AI 角色回复。
- 每条消息都能清楚显示发送者身份。
- AI 角色回复能体现各自的人设或回复风格差异。
- 刷新页面或重新进入群聊后，历史消息仍然存在。
- 没有实现 BACKLOG 中的非 P0 功能。

## 6. 本轮明确不实现的功能

- 用户登录与多用户账号隔离。
- 完整 API 管理后台。
- AI 角色删除、编辑和发言权限开关。
- 稳定实时消息推送体验。
- AI 角色之间的补充或反驳。
- 群聊主题引导。
- 思考强度配置。
- Token 花费统计面板。
- 文件上传与 AI 文件分析。
- 工具调用。
- 白天和黑夜模式。
- 公开社区或多人实时协作聊天室。
- 插件或角色市场。
- 模型训练、微调或模型托管。

## 7. 技术栈要求

- 后端必须使用 Go 语言。
- 不允许自行替换为 Python、Node.js、Ruby、Java 或其他非 Go 后端技术栈。
- 必须提供网页端用户界面，不能把 v0.1 做成纯后端服务或命令行 demo。
- 必须具备持久化能力，用于保存群聊、AI 角色、模型 API 配置和消息历史。
- 必须接入真实模型调用路径；可以在模型服务不可用时暴露清晰错误，但不能用纯 mock 冒充 MVP 已完成。
- 前端技术选型、Go Web 框架、数据库类型和模型供应商在实现前需要确认；未确认前不得擅自扩展或替换为超出本合同的技术路线。

## 8. 验收清单

- [ ] 可以从网页端创建群聊并填写群聊名称。
- [ ] 可以从网页端进入群聊详情页。
- [ ] 可以从网页端添加至少两个 AI 角色。
- [ ] 每个 AI 角色可以配置名称、人设、回复风格和模型。
- [ ] 可以从网页端填写基础模型 API 配置。
- [ ] 可以在群聊中发送用户消息。
- [ ] 至少两个 AI 角色会基于用户消息分别回复。
- [ ] 用户消息和 AI 角色消息都显示清晰发送者身份。
- [ ] 群聊消息会持久化保存。
- [ ] 重新进入群聊后能看到历史消息。
- [ ] AI 回复不是固定 mock 文本伪造的闭环。
- [ ] 未实现本合同明确排除的非 P0 功能。
- [ ] 实现过程中没有自行替换技术栈。

## 9. 停止条件

遇到以下情况必须停止并说明，不能擅自降级或改范围：

- 无法在当前技术栈约束下完成网页端用户操作路径。
- 无法使用 Go 作为后端技术栈。
- 无法提供持久化能力。
- 无法接入真实模型调用路径，只能做纯 mock 闭环。
- 实现成本与本文档要求冲突。
- 必须引入 BACKLOG 中的非 P0 功能才能继续。
- 需要在 P0 范围外新增登录、完整 API 管理、工具调用、文件上传、Token 统计或实时推送等能力。
- 模型供应商、API Key 存储方式、数据库或前端形态必须先由用户确认。

## 10. 如果遇到非 MVP 需求，如何处理

- 不直接实现。
- 记录到 `docs/ai/BACKLOG.md` 的对应 P1、P2 或 P3 分组。
- 如果该需求会改变 v0.1 的核心验收标准，先停止并请求用户确认。
- 如果该需求只是优化体验，不影响 v0.1 闭环，保留为后续版本，不进入本轮实现。
- 如果该需求与 Non-goals 冲突，明确标记为不做。
- 每轮结束必须同步更新 `docs/ai/STATUS.md`、`docs/ai/NEXT.md` 和 `docs/ai/BACKLOG.md`。

## 人工确认

在开始写 v0.1 业务代码之前，用户必须人工确认本文档可以作为 MVP 实现合同。
